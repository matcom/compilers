## Bloque 3 — Parser

**Recursive-descent + Pratt precedence climbing.** `src/parser/Parser.cpp` (883 líneas) implementa un parser manual descendente. La disciplina general es RD con `check`/`match`/`expect`, pero las expresiones binarias usan **Pratt (precedence climbing)** en `parseBinaryOp(minPrecedence)`. Vale destacar la contradicción con el REPORT: el estudiante etiqueta al parser como "LL(1)" en la introducción (sección 4) pero luego en 4.2 admite que las expresiones se resuelven con "escalada de precedencia" — el código coincide con lo segundo.

**Niveles de precedencia** (`Parser.cpp:56-69`):
- 1 → `OR` (`|`)
- 2 → `AND` (`&`)
- 3 → `EQ`, `NE`
- 4 → `LESS`, `GREATER`, `LE`, `GE`
- 5 → `PLUS`, `MINUS`
- 6 → `STAR`, `SLASH`, `MOD`, `AT`, `ATAT`
- 7 → `CARET` (asociativo por la derecha)

**Anotaciones de tipo.** `parseTypeAnnotation()` acepta `:` seguido de identificador con corchetes anidados como sintaxis para `Vector[T]` (`Parser.cpp:82-100`). **No** acepta `(T) -> U` (tipos función), lo que impide sintaxis de lambdas y functors.

**Desambiguación `{...}` como bloque vs arreglo.** El parser hace lookahead sobre el primer token dentro de las llaves para decidir si es un bloque de expresiones (separadas por `;`) o un literal de arreglo (separadas por `,`). Esta lógica se ubica en `parsePrimary` cuando encuentra `LBRACE`.

**Sintaxis específica:**
- `new Type[size]{i -> expr}` para crear arreglos inicializados con expresión indexada. Nodo `NewArrayNode`.
- `[e1, e2, ...]` para literales de vector.
- `[expr | var in iterable]` para comprensión de vector con `OR` como delimitador (verificado en `Parser.cpp:114`).
- Miembros: `.attr`, `obj.method(...)`, `x[i]`, `x[i] := v`.

**Recuperación.** El parser acumula todos los errores sintácticos en `errorMessages` con formato `(línea,columna) SYNTACTIC: <msg>` (`Parser.cpp:49-54`) y devuelve nodos por defecto para continuar. `main.cpp:323-325` los reporta y sale con código 2.

## Bloque 4 — Análisis semántico

**Dos pasadas: TypeInference + TypeChecker.** El análisis semántico se divide en dos fases secuenciales sobre el mismo AST:

1. **TypeInference** (`src/semantic/TypeInference.cpp`, 706 líneas): recorre el AST recolectando usos de parámetros no anotados (`collectUsage`) y decide un tipo por defecto (`inferParamType`) según cómo se usa el parámetro (aritmética → `Number`, concatenación → `String`, uso booleano → `Boolean`, llamadas a métodos → tipo cuyo método coincida). También propaga tipos desde el llamador cuando la firma del callee permite ajustarse.

2. **TypeChecker** (`src/semantic/TypeChecker.cpp`, 1145 líneas): verifica compatibilidad de tipos, resolución de nombres, acceso a atributos y métodos, herencia bien formada, sobreescritura correcta. Utiliza el `TypeRegistry` (`src/semantic/SymbolTable.cpp`).

**TypeRegistry** (`SymbolTable.cpp:4-43`). Los tipos built-in registrados son: `Object` (raíz), `Number`, `String`, `Boolean` (los tres con `parentName="Object"`), y `Vector` con dos métodos precomputados `size(): Number` y `append(Object): Number`. `Vector` es tratado como paramétrico vía `TypeDescriptor.typeArgs` en el chequeo, pero en el registro es un tipo built-in único.

**Subtipado** (`SymbolTable.cpp:59-90`). `isSubtypeOf(t1, t2)` implementa: reflexividad, `Object` como top, tratamiento especial de `Vector`, restricción de built-ins solo a sí mismos y a `Object`, y camino ascendente por `parentName` con detección de ciclos (`std::set<std::string> visited`). Esta detección de ciclos es una fortaleza real del código.

**LCA (Lowest Common Ancestor)** (`SymbolTable.cpp:96-125`). Usado para inferir el tipo de expresiones `if` (ramas con tipos diferentes se unifican en su ancestro común) y para el tipo de elementos de un vector literal. La implementación camina la cadena de `t1` buscando un ancestro de `t2`, con `Object` como fallback.

**Regla de `null`.** `NullLiteralNode` tiene tipo `Object` (`TypeChecker.cpp` maneja el nodo con este tipo). Se permite comparar `null` con cualquier objeto vía `==`/`!=`, pero **no** se permite `usuario_defined == usuario_defined` — solo built-ins y comparaciones con `null` son admisibles. Esto es coherente con la semántica de HULK.

**Atributos privados.** El TypeChecker verifica en el momento del `MemberAccess` que el atributo o bien pertenece al tipo actual o bien es heredado por la cadena de padres. `hasOwnAttribute` y `findAttribute` (`SymbolTable.cpp:152-183`) recorren la cadena de herencia con detección de ciclos.

**`for` restringido.** El `for` solo se permite sobre `range(...)` o sobre valores de tipo `Vector[T]` — no hay soporte de iterables/protocolos generales, coherente con la ausencia de features de protocolos.

**Errores semánticos.** Se acumulan en un vector; `main.cpp:353-362` los reporta con formato `(línea,columna) SEMANTIC: <msg>` y sale con código 3.

## Bloque 5 — Generación de código y VM

### Backend BANNER + VM

**BANNER como IR lineal.** `CodeGenerator` emite instrucciones textuales de tres direcciones, similar a un ensamblador. Ejemplos: `ADD t0 t1 t2`, `MUL t0 t1 t2`, `VCALL`, `ALLOCATE`, `GETATTR`, `SETATTR`, `IS_OP`, `AS_OP`, `VNEW`, `PRINT`. La lista completa está en `include/vm/VM.hpp:9-32` como `enum class OpCode` con ~40 opcodes.

**TypeMeta y layout.** `CodeGenerator` computa `TypeMeta` por tipo: layout de atributos (offset por nombre) y vtable de métodos (`__type_TypeName_methodName` como convención de nombres). Los tipos se procesan por **ordenamiento topológico** para garantizar que los padres se registren antes que los hijos y que el layout se pueda extender.

**VM.** `src/vm/VM.cpp` (1393 líneas) es un intérprete de stack. Usa **arreglos paralelos** para objetos heap: uno para dobles, uno para strings, uno para handles (referencias a otros objetos), uno para booleanos. `VectorObject` sigue la misma estructura para elementos heterogéneos. `HulkValue` es una unión etiquetada con `HULK_NUM`, `HULK_STR`, `HULK_BOOL`, `HULK_OBJ`, `HULK_NUL`, `HULK_VEC`.

**Opcodes clave:**
- `AND_OP` en la VM es un opcode de doble propósito: si algún operando es string, hace concatenación; si son numéricos, aplica el `&` lógico. Esto es raro y sugiere una historia de la implementación (probablemente se reutilizó el mismo opcode antes de separar `@`/`@@`).
- `VCALL` para llamadas a métodos virtuales — busca en la vtable del tipo del receptor.
- `ALLOCATE` reserva un objeto con el layout de un tipo dado.
- `GETATTR`/`SETATTR` acceden a atributos por nombre resuelto en tiempo de codegen (offset).
- `IS_OP`/`AS_OP` implementan el chequeo/conversión de tipo caminando la cadena de padres.

### Backend C

**Runtime embebido como string literal** (`CCodeGenerator.cpp:11-151`). El generador escribe el runtime completo al inicio de `output.c`, ~140 líneas de C que incluyen:

- `HulkValue` tagged union con enum `HulkType` (`NUM`, `STR`, `BOOL`, `OBJ`, `NUL`, `VEC`).
- Aritmética envolvente: `hulk_add`, `hulk_sub`, `hulk_mul`, `hulk_div`, `hulk_pow`, `hulk_mod`.
- Comparaciones envolventes: `hulk_lt`, `hulk_le`, `hulk_gt`, `hulk_ge`, `hulk_eq`, `hulk_ne`.
- Concatenación: `hulk_concat` (para `@`).
- Vectores: `hulk_vec_new`, `hulk_vec_get`, `hulk_vec_set`, `hulk_vec_size`, `hulk_vec_append`.
- Alocación: `hulk_alloc` (= `malloc`).
- RTTI: `hulk_is_type` que recorre `_typeParents[]` — un arreglo global emitido por el codegen que codifica la cadena de herencia por índice de tipo.
- Impresión: `hulk_print`.

**`is`/`as`.** `IsNode` se traduce a `hulk_is_type(obj, TypeID)` que retorna 1/0. `AsNode` hace el chequeo y, si falla, `printf` de error + `exit(1)` — semántica de fallo dinámico. Ambos siguen el patrón esperado.

**Vtables como arreglos globales.** Cada tipo definido por el usuario obtiene un arreglo global de punteros a función; los métodos virtuales se llaman resolviendo por índice en la vtable a través del type-id embebido en el objeto.

**Compilación final.** `main.cpp:395` invoca `gcc -O2 output.c -o output -lm` y limpia (o mantiene con `--keep-c`) el `.c` intermedio.

## Bloque 8 — Diagnóstico de fallas principales

**Categorías con fallos** (todas no marcadas por el estudiante):

### `ok/interfaces/` — 0/6 tests pasan, fallo LEXICAL

Los tests usan la keyword `interface`:
```hulk

    to_string(): String;

```

El lexer registra `protocol` → `KW_PROTOCOL` (`Lexer.cpp:18`) pero **no reconoce `interface`**. Al encontrar el identificador `interface` seguido de `Printable`, el parser ve `IDENTIFIER IDENTIFIER LBRACE` y falla con SYNTACTIC. Incluso si se cambiara a `protocol`, el parser tampoco tiene una regla que consuma `KW_PROTOCOL` — es una **keyword muerta**. Faltan tanto lexer support como parser rule + AST node + semantic + codegen.

### `ok/lambdas/` — 0/10 tests pasan, fallo SYNTACTIC

Los tests usan:
```hulk

```

Dos ausencias:
1. **Tipo función `(T) -> U`** no está en `parseTypeAnnotation` (`Parser.cpp:82-100`) — solo acepta `IDENTIFIER` con corchetes de vector.
2. **Expresión lambda `function (...) -> expr`** no está en `parsePrimary`. El keyword `function` solo se admite como declaración top-level de función.

Falta también el nodo AST `LambdaNode` y todo el soporte semántico/codegen para clausuras.

### `ok/macros/` — 0/8 tests pasan, fallo LEXICAL/SYNTACTIC

El lexer registra `def` → `KW_DEF` (`Lexer.cpp:19`) pero **no `define`**. Los tests con `define` se atragantan con `IDENTIFIER IDENTIFIER LPAREN` en el parser. Y `KW_DEF` es otra keyword muerta: el parser nunca la consume, no hay `DefineNode` ni expansión de macros.
